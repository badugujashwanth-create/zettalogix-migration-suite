# NARRATION

Welcome to Zettalogix Migration Suite version zero point three. This is a frontend and Electron control-plane prototype for planning and inspecting content-migration work. The yellow boundary banner is deliberate: every record in this walkthrough is fictional, the browser state is local, and no migration, identity, Google Drive, SharePoint, or external API service is connected.

The command center summarizes the supplied job and connection state. Search spans jobs, sites, libraries, connections, and help topics without leaving the workspace. The running and at-risk indicators come from the same in-memory dataset shown below. They are operational views of demo evidence, not claims that a real worker is processing content.

The migration ledger separates queue metrics, operator actions, and the job table. Existing records demonstrate a running finance wave and a completed human-resources archive. The interface labels its polling source as deterministic in-memory state during this walkthrough, so a reviewer can distinguish a genuine UI transition from external migration execution.

The new-migration control opens a four-step blueprint covering discovery, source selection, destination selection, and review. In demo mode, the wizard explicitly says it uses fictional connections and contacts no provider. This scene demonstrates the workflow structure; it does not imply that the separately owned backend discovered content or validated production credentials.

Next, the guided simulation creates a legal-records draft directly through the same local store used by the interface. The draft contains an inspectable source path, destination site, library, folder, and metadata policy. Creation adds a visible notification and ledger entry, proving the application state changed without sending a network request.

Starting the draft moves it into the running state, assigns deterministic progress, and records a synthetic start event. This is intentionally a state-machine demonstration. The control is disabled when a job is already running, preventing duplicate starts in both the walkthrough and the live client while leaving real execution authority with the external backend.

The pause transition updates that same job and prepends an operator-visible event. Notifications are shown long enough to confirm each action and then cleared before the next scene so evidence does not obscure the interface. Reset demo restores the original two-job baseline, making the walkthrough repeatable for recruiters and reviewers.

Migration readiness is evidence-derived. Five checks inspect the referenced source, referenced target, source-to-destination mapping, failure pressure, and available event history. Ready, review, or blocked status follows only from those fields. The score is not predictive artificial intelligence and is not a promise that a real migration will succeed.

The connection inventory contains a fictional file-share source and Microsoft 365 destination. Demo fields stay in memory and the Google Picker is not loaded. The interface can still exercise local registration rules, display endpoint types, and provide clear input guidance while provider authentication and connection execution remain outside this repository.

Testing a synthetic connection changes the selected record to healthy and records the local timestamp and message. No Google, Microsoft, Supabase, or migration request is made. This boundary is covered by automated tests and reinforced by the visible network-off label throughout the guided simulation.

Execution defaults are browser-only planning values in demo mode. Concurrency, retry count, failure notifications, and telemetry preference are saved locally, but the screen does not claim they control an absent worker. The live client likewise describes these as client defaults that require compatible backend support before they can affect execution.

The help center collects setup requirements, failure guidance, and links to official provider documentation. It also keeps the ownership model explicit: this repository contains the React client and hardened Electron shell, while the migration API and worker belong to a separate external repository that was not copied, modified, or production-verified here.

The simulation is complete. It covered search, migration monitoring, a four-step blueprint, create, start, pause, evidence-derived readiness, connection inventory and testing, browser-only settings, help guidance, reset, and CSV export. Version zero point three is verified as a network-free synthetic control-plane demo, not as proof of production migrations or an owned backend.
